## Tavolo di Lavoro — Piano di sviluppo TODO 2026-04-09

### Posizioni

**Architetto (ARCH)**
Separare subito la migrazione architetturale delle unità di misura (punto 4) dalle fix UI, trattandola come "Fase 0" fondativa. Le 5 fix simulazioni e il grafico dipendono indirettamente dall'esito della conversione nutrienti; farle in sequenza sbagliata crea rework. L'assunzione critica da non dare per scontata è lo stato dei dati storici: chi ha caricato nutrienti manualmente con unità inconsistenti? Serve un audit dei dati prima di rendere l'unità canonica vincolante. Raccomanda: migration + factory conversione + test come PR separata e prerequisita.

**Backend (BE)**
Il blocco critico è la normalizzazione DB nutrienti: la tabella `Nutrient` ha `DefaultUnitOfMeasureId` ma non un campo `UnitaMisura` canonica esplicita. L'endpoint AI vision (punto 3) è il più rischioso: senza rate limiting per utente i costi LLM esplodono. Propone una tabella `NutrientExtractionCache(imageHash, extractedNutrients, confidenceScore)` per evitare estrazioni duplicate. Per le preferenze utente grafico suggerisce colonna JSON su `AspNetUsers`. Ordine commit: Fase 0 → 1 → 2 → 3, ogni commit deve essere shippabile.

**Interface (UI)**
Ha già individuato i file esatti. `useSectionConfigs.ts` restituisce già `DisplayOrder` ma non lo usa per ordinare: basta aggiungere `.sort()`. La fusione alimenti+piatti richiede un computed `combinedFoodDishOptions` che elimina il radio button "Tipo". Il formatter `formatNutrient` con `toFixed(2)` è duplicato in `DailySimulationDetail.vue` e `DailySimulationCompareModal.vue`: va estratto in un'utility condivisa con `toFixed(3)`. Per il grafico, la riga 66 di `DailySimulationChart.vue` inizializza un Set vuoto: basta popolarlo con i 5 nutrienti default.

**User Experience (UX)**
L'utente di un'app nutrizionale vuole aggiungere cibi velocemente senza errori, non aspetta miracoli dall'AI. Le fondamenta invisibili (UoM canoniche, dati corretti) vengono prima della magia. Il rischio maggiore è l'AI senza trasparenza: se la conversione di unità non è mostrata esplicitamente, l'utente non verifica e si genera garbage-in garbage-out, fatale in nutrizione. Priorità: simulazioni + fondamenta UoM → AI con verifica → grafico e precisione.

### Punti di Tensione

- **Ordine Fase 0 vs Quick Wins**: ARCH e BE insistono che la migration UoM venga prima di tutto; UX vorrebbe le fix simulazioni subito per valore immediato. Soluzione: le fix UI pure (titolo, layout, ordinamento sezioni, formatter, grafico default) non toccano il DB e possono andare in Commit 1 sicuro. La migration UoM è Commit 2 separato.
- **Complessità AI vs ROI**: BE segnala rate limiting e caching come necessari sin dall'inizio; UX vuole la funzione disponibile presto con la trasparenza sulla conversione come requisito non negoziabile. Soluzione: l'endpoint AI è Fase 4, ma il design include rate limiting e mostra conversioni esplicitamente dalla prima iterazione.
- **Formatter centralizzato vs patch locale**: UI vuole un'utility condivisa; le altre discipline sono indifferenti. Soluzione: va fatto, zero rischio, incluso in Commit 1.

---

### Raccomandazione

Eseguire in 5 commit progressivi, ognuno shippabile e verificabile indipendentemente. La scelta dipende dalla tolleranza al rischio: se i dati esistenti sono già consistenti, la migration UoM può andare in Commit 2 senza rischi; se ci sono dati storici con unità miste, fare prima un audit SQL e un batch fix come pre-requisito non tracciato nei commit sottostanti.

### Commit 3 — Unità di misura canoniche (DB + backend)

**Scope**: modifiche strutturali al DB e alla logica di inserimento nutrienti.

**Decisione già presa nel TODO**: unità canonica fissa per nutriente, conversione all'inserimento. Alternativa normalizzazione a query-time scartata.

#### 3a. Migration SQL

```sql
-- Aggiunge colonna UnitaMisura alla tabella Nutrient
ALTER TABLE [Nutrient] ADD [UnitaMisura] NVARCHAR(20) NULL;
-- (poi popolare con valori default per nutrienti esistenti)
UPDATE [Nutrient] SET [UnitaMisura] = uom.[Abbreviation]
FROM [Nutrient] n
JOIN [UnitOfMeasure] uom ON n.[DefaultUnitOfMeasureId] = uom.[Id]
WHERE n.[UnitaMisura] IS NULL;
-- Rendere NOT NULL dopo il populate
ALTER TABLE [Nutrient] ALTER COLUMN [UnitaMisura] NVARCHAR(20) NOT NULL;
```

// Posizionamento suggerito: src/Dr.NutrizioNino.Api/Services/UnitConversionService.cs
static decimal Convert(decimal value, string fromUnit, string toUnit) => (fromUnit, toUnit) switch {
    ("mg", "µg") => value * 1000,
    ("µg", "mg") => value / 1000,
    ("g", "mg")  => value * 1000,
    ("mg", "g")  => value / 1000,
    _ when fromUnit == toUnit => value,
